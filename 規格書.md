## 一、專案概念

網頁版「歷史戰役 3D 推演劇場」:

- 3D 戰場 + 時間軸驅動的戰況推演,體驗如「有導播的歷史紀錄片」
- 首頁為串流平台式「節目單」,海報卡片選擇戰役
- 一套引擎多場戰役:戰役資料(JSON + 素材)與引擎完全分離,新增戰役只需新增資料夾
- 後續用途:YouTube 頻道影片素材來源(運鏡考慮 16:9 錄製、UI 可隱藏)
- 第一場戰役:**赤壁之戰(演義版)**

## 二、技術定調

- 純前端、無 build step
- Three.js **r160**,jsdelivr CDN + importmap ES module,含 OrbitControls
- 程序化地形(高斯山丘高度場 + 頂點著色 + canvas 程序化貼圖),不使用 DEM
- 引擎 v1 即支援:陸地高度場 + **水面平面**(水戰為一等公民)
- 單位類型:`infantry`(陸軍方陣)/ `fleet`(船隊)
- 驗收方式:local server(如 `python -m http.server`)瀏覽器驗收
- 介面預設繁體中文,支援多語系(見第四節)

## 四、全域約定

### 座標系
- 世界單位平面座標 `[x, z]`,原點在戰場中心,**x = 東、z = 南**
- 心理比例 1 單位 ≈ 10 公尺(不嚴格)
- 赤壁戰場約 `1200 × 800`

### 時間系
- **章節制**:每章有獨立時間軸,事件 `t` = 章內分鐘偏移(number)
- 每章 `playback_sec` 定義 1× 速度下的實播秒數 → 各章自帶快轉倍率(鋪陳章大步快轉、決戰夜細刻度)
- 畫面顯示時間用 `time_display` 自由字串(支援「二更」等時辰),與排程解耦
- UI 時間軸為一條全域軸,以章節分段標記,可拖曳跨章、點章節名跳轉

### 多語系
- 支援:`zh-TW`(預設)/ `zh-CN` / `en` / `ja`
- 引擎 UI 字串:`engine/i18n.js` 內建四語字典
- 戰役內容字串:**內聯多語物件** `{"zh-TW":"…","zh-CN":"…","en":"…","ja":"…"}`
  - 引擎以 `pick(obj)` 解析,fallback 鏈:目標語系 → zh-TW → 物件第一個值
  - 純字串視為單語(等同只有 zh-TW)
- 語言切換器:節目單頁右上角 + 推演頁設定區;記 localStorage;`?lang=` 參數優先(錄影固定語言用)
- 演義 `quote` 原文:**任何語系都顯示漢文原句**(風味元素),en/ja 可選填譯文小字附於原句下

```json
        }
```


#### 事件 type 語意
| type | 必要欄位 | 語意 |
|---|---|---|
| `move` | unit, to, (t_end) | 單位自 t 起移動至 `to`,t_end 抵達(省略則瞬時) |
| `engage` | unit(s) | 進入交戰狀態;可帶 `losses`(兵力插值目標)|
| `fire` | unit / units | 指定對象起火(腳本驅動延燒:多個 fire 事件分批)|
| `defect` | unit | 詐降/倒戈,名牌閃爍 defect 徽章 |
| `rout` | unit | 潰走狀態,名牌淡化 |
| `status` | unit | 通用狀態/兵力修正(`troops`、`badge`)|
| `narration` | title/desc | 純旁白字卡(可帶 speaker/quote/fx)|
| `camera` | camera_hint/pos | 純鏡頭指令 |

- `camera_hint`:`follow / overview / none`(省略 = 導播自行判斷)
- `fx`(選填):事件附帶特效,`kind`:`volley / explosion / smoke / ring …`
- `speaker`(選填):commanders id → 字卡左側顯示頭像(3:4)+ 名字

## 六、執行期模型(重要)

**單位狀態 = 時間的純函數**(參考站驗證的設計):

1. 載入時,timeline.js 將 `move / rout / status / engage / fire` 事件**編譯為每單位的關鍵幀軌道**:位置 keys、兵力 keys、狀態 keys、透明度 keys
2. 執行期一律 `posAt(unit, t)` / `troopsAt(unit, t)` / `stateAt(unit, t)` 純函數取值
3. 因此時間軸可任意拖曳、倒帶、跳章,全場狀態永遠正確,且每次播放完全確定(對 YouTube 錄製重要)
4. `narration / camera` 類事件不進軌道,屬「表演層」,seek 時只取目前區間最後一張字卡
5. 章節在載入時銜接成一條全域播放軸(各章依 `playback_sec` 換算播放速率)

## 七、引擎功能清單

### 操作(比照參考站)
- 拖曳旋轉 / 滾輪縮放 / 右鍵平移(OrbitControls)
- Space 播放暫停、0.5×/1×/2×/4× 速度、時間軸拖曳、←→ 跳時間、章節跳轉
- 點擊部隊名牌 → 鏡頭飛往該隊;hover → tooltip(部隊介紹、兵力)

### 自動導播
- camera state machine:事件觸發鏡頭飛往現場(`camera_hint`)
- 使用者手動操作即暫停自動運鏡(manual hold),下一事件時導播接管

### UI 層(DOM overlay,worldToPx 投影)
- 部隊名牌:陣營色邊條 + 名字 + 兵力(隨 t 插值遞減)+ 狀態徽章
  - 徽章:進軍 / 交戰 / 潰走 / **詐降(閃爍)** / 撤離 / 本陣
- 地名標籤:依 `landmarks.type` 分樣式
- 旁白字卡(lower-third):title/desc + 選填 speaker 頭像 + 選填 quote 原文(小字)
- **war-meter**:雙方(按 side)總兵力即時數字 + 比例條
- 片頭 title screen、結算 end card(勝負、交戰時間、雙方損失、關鍵轉折、「↺ 重新觀看」)
- 電影感 overlay:膠片顆粒 + 暗角 + letterbox 黑邊(cinema mode)
- 語言切換器

### 大氣系統
- `skyColor(t)` / `fogDensity(t)` 隨模擬時間變化(赤壁:夜 → 黎明)
- 光照、霧、天色與章節時刻聯動

### 特效(effects.js)
- 火攻:**腳本驅動延燒**(fire 事件分批指定),GPU Points 粒子火焰 + 煙柱 + 水面火光反光,**粒子上限 1~2 萬**
- 箭雨曳光 `volley`(草船借箭)
- 煙、衝擊環等通用 fx

### 音效(後做)
- Web Audio 全合成,無音檔

### 錄影友善模式
- 一鍵隱藏 UI、鎖 16:9,與 letterbox 互補(優先度低)

## 八、美術方向

- **棋盤推演風**:部隊 = 簡潔幾何體陣列(步兵 = 方陣方塊群、船隊 = 低面數船形體排列,instanced)+ 陣營色旗幟(canvas 貼圖)
- 質感靠燈光、水面反射、火特效與電影感 overlay,不走寫實路線
- 地形:頂點著色 + canvas 程序化貼圖並用(長江岸線、水寨範圍畫進貼圖)
- 主帥肖像(使用者提供):**3:4 直式、胸像、去背 PNG、兩軍視線相對(一左一右)**,AI 生成統一畫風

## 九、第一場戰役:赤壁之戰(演義版)

### 已定事項
- 依《三國演義》敘事,介面標註「本推演依《三國演義》敘事」
- 地理採蒲圻說(今湖北赤壁市),長江江面為主戰場,北岸烏林為曹軍水寨

### 章節分鏡
1. **第一章 風起江上**(快轉,約數日):蔣幹盜書 → 草船借箭 → 苦肉計 → 借東風

3. **尾聲 華容道**(鏡頭轉陸路):曹操敗走 → 三道伏兵 → 關羽義釋 → 結算畫面

### 內容備註
- quote 原文引句只放名場面(借東風、火燒連環船、華容道等),不是每事件都有
- 四語內容:初稿先寫 zh-TW,en/ja 譯文由使用者最終人工校對

### Phase 6 美術升級規格(6-A ~ 6-F)

> 定案日期:2026-06-11。緣起:與參考實作 https://github.com/yazelin/red-cliffs-3d 對比後,
> 確認廉價感根因為「缺乏整體美術指導」而非個別技術缺失。
> 唯一標準:**好看、不廉價**。參考實作的手法可用,但每項均須經過取捨判斷,不照抄參數。

#### 總則(6-A ~ 6-F 全部適用)

**美術身份**:一座精緻沙盤,被紀錄片攝影機拍攝。**單一統一色盤**,僅以「時刻」(晝/夜/霧)變化氛圍;不做章節換風格。

**兩條鐵律**(所有技術決策服務它們):

1. **世界低飽和**:地形、水、天全部壓進灰調大地色家族;畫面中飽和色僅限三樣——陣營色(旗幟/名牌)、火光、晨昏天色。
2. **敢暗**:夜就是真的黑,火是場上唯一暖光源並真實照亮江面;白天靠暖日光/冷天光的明暗對比給體積感。

**色盤草案**:白天=灰藍天頂、暖灰地平線、橄欖卡其地形、暗藍綠江面;夜=深藍黑天、近黑江面;霧=吃掉地平線的灰白。晝夜為**整套色盤插值**,由 atmosphere 關鍵幀驅動,不是只調太陽亮度。

**硬性約束**:

- 確定性不可破壞(規格書第六節):所有動態視覺(水波、旗飄、火光搖曳)一律由**全域播放進度 p** 驅動,不得用 wall-clock 累加或執行期隨機;噪聲一律固定 seed。
- UI DOM overlay(名牌/字卡/war-meter/時刻面板)螢幕佔比與字級不變。
- FPS 無明顯下降;不增加每幀 draw call 數量級。
- 每階段獨立 commit,瀏覽器驗收後才進下一階段(6-A 是地基,必須最先;6-C 在 6-D 之前;其餘順序建議照字母序)。

**共同驗收(每階段)**:完成後截三張「劇照」——白天 overview、夜戰 follow、濃霧(草船借箭段)——與前一階段對照需可見改善,且「像劇照,不像 tech demo」;`.verify_phase5.py` 通過;console 無錯誤;任意 seek/倒帶後畫面與重播一致。

#### 6-A 全域調色 + 光照結構(地基)

- **檔案**:`engine/terrain.js`(高度色階、貼圖基底)、`engine/scene.js`(燈光、曝光)、`battles/chibi/events.json`(既有 atmosphere 關鍵幀色值連動)
- **做法**:
  1. 地形高度色階 `stops` 整套降飽和、壓亮度,向橄欖/卡其灰調靠(現值 `0x7d9454`、`0xb3a275` 等飽和度過高,是「塑膠草皮感」主因)。
  2. 程序化貼圖基底自近白(224)改中灰,確保與頂點色相乘後不過曝;田野色塊/岸線色同步重調。
  3. 水面純色暫調更深的暗藍綠(6-C 換 shader 前的過渡)。
  4. 光照結構:hemi 0.9 → 約 0.45~0.6(現值把陰影面填平,等於沒有體積);sun 維持暖色,與 hemi 冷色形成冷暖對比;`toneMappingExposure` 1.1 隨整體重調。
  5. 既有 atmosphere 關鍵幀色值(events.json)按新色盤等比修正,避免章節切換跳回舊色。
- **驗收**:截圖對比——整體飽和度/亮度下降、山體明暗面對比可辨;三章 atmosphere 過渡無突兀色跳;UI 不變。

#### 6-B 漸層天穹 + FogExp2

- **檔案**:`engine/scene.js`(skydome、霧)、`engine/atmosphere.js`(schema 擴充)、`battles/chibi/events.json`(atmosphere 資料遷移)
- **做法**:
  1. **skydome**:大半球 + ShaderMaterial,天頂色/地平線色兩色 `pow` 曲線漸層;uniform 由 atmosphere 每幀驅動,取代 `scene.background` 純色(廉價感第一大來源)。
  2. **霧**:`THREE.Fog`(線性 3000/7600,在 maxDistance 2400 的常用視距內幾乎無作用)→ `THREE.FogExp2`;密度量級初估 0.0003~0.001,按 2400×1600 地形實調(參考實作 680×460 地形用 0.0018~0.0026,我們尺度約 3.5 倍,密度近似反比縮放)。霧色與地平線色一致,遠景自然溶接。
  3. **atmosphere schema 擴充**:`sky`(單色)→ `sky_top` / `sky_horizon`;`fog: [near, far]` → `fog_density`。**一次性遷移** events.json 既有關鍵幀,不留雙軌相容碼。
  4. **霧氣即演出**:草船借箭段關鍵幀加密(濃霧鎖江,劇情元素),平時維持薄霧空氣感——同一套系統,不是另一種風格。
- **驗收**:地平線不再是純色切線;遠景溶入霧;草船借箭段濃霧可辨且霧散後恢復;拖曳/跳章天色霧效正確插值。

#### 6-C 水面 shader

- **檔案**:`engine/terrain.js`(水面材質)、`engine/atmosphere.js`(水色深/淺兩色驅動)、`engine/main.js`(uTime 接全域播放進度)
- **做法**:
  1. 水面 `MeshStandardMaterial`(靜止純色,佔畫面最大面積卻一動不動,是畫面「死」的主因)→ `ShaderMaterial`:
     - vertex:2~3 層不同頻率/方向的 sine 疊加微幅波動(視覺微動即可,不做寫實浪)。
     - fragment:以波高混合 `uDeep` / `uLit` 兩色;`transparent`、opacity ≈ 0.9。
  2. **uniform 約定**:`uTime` 由全域播放進度 p 換算(確定性鐵律,禁 wall-clock);`uDeep` / `uLit` 由 atmosphere 驅動(晝夜換水色,schema 的 `water` 欄位擴成兩色或由單色衍生);`uGlow`(火光映江強度,本階段先留接口,6-D 接入)。
  3. 既有 `applyEdgeFade` 邊緣迷霧邏輯移植進自訂 shader(自訂 ShaderMaterial 不走 onBeforeCompile 鏈)。
- **驗收**:江面常時微動、有深淺變化;同一 p 值波形完全一致(seek 確定性);FPS 無明顯下降。

#### 6-D 夜戰光源(火 = 光)

- **檔案**:`engine/effects.js`(火光點光源、餵 uGlow)、`battles/chibi/events.json`(第二章夜間關鍵幀壓黑)
- **做法**:
  1. 燃燒單位/位置掛 `PointLight`(暖橘 0xff7a26 系,distance 按單位佔地、強度隨火勢)。**上限管理**:同時點光源 ≤ 4~6,多處起火時就近合併,避免光源數失控。
  2. 光強隨 p 確定性計算:火勢自 fire 軌道取值;搖曳用固定相位 sine 疊加,不用隨機。
  3. 水面映火:按起火點與火勢餵 6-C 的 `uGlow`,江面映出橘紅。
  4. 第二章夜間 atmosphere 關鍵幀真正壓黑(ambient 量級 0.1~0.2、天/霧近黑),讓火成為畫面主光源——「黑夜裡飄著不發光的粒子」是目前夜戰最致命的廉價感。
- **驗收**:夜戰段落以火光為主光源、江面映紅;火滅後無殘留光源;倒帶/seek 火光狀態一致;非夜戰章節不受影響。

#### 6-E 地形 fbm 細節

- **檔案**:`engine/terrain.js`
- **做法**:
  1. `heightAt()` 疊加 2~3 octave 連貫值噪聲(幅度約山丘高度 5~10%、波長數百世界單位起):高斯山丘輪廓保留(戰役資料語意不變),消除「光滑饅頭」感;水域以下不擾動或大幅衰減,避免岸線破碎。
  2. 頂點亮度抖動:逐頂點白噪 `hash01`(椒鹽顆粒感)→ 低頻連貫噪聲色塊(田野/植被斑塊感);貼圖 `fieldNoise` 同步重審,避免兩層噪聲打架。
  3. 噪聲固定 seed,每次載入一致。
  4. 連動驗證:單位貼地、樹木散佈、營地/道路貼圖都取樣 `heightAt`,改動後自動一致,但需逐一檢查岸線、華容道路徑、樹木無穿幫。
- **驗收**:山體有自然起伏;岸線/路徑/樹木無穿幫;兩次載入畫面一致。

#### 6-F 旗幟飄動 + 風力參數

- **檔案**:`engine/units.js`(旗面幾何/材質/shader)、`engine/main.js`(uTime/uWind 驅動)、`battles/chibi/events.json`(借東風段風力)
- **做法**:
  1. 旗面 `FLAG_GEO = PlaneGeometry(7, 4.2)`(單段,無法變形)→ 多段(如 8×3 段);vertex shader 按「距旗桿距離」比例 sine 波動(旗根不動、旗梢擺幅最大);船帆若做,振幅減半。
  2. 旗面材質 `MeshBasicMaterial` 不受光,夜間壓黑後會變懸浮亮片(6-D 連動問題)→ 改受光材質,或 shader 內乘環境亮度係數。
  3. **uWind 風力參數**:預設 1.0;借東風後 → 約 2.5(波速與振幅同步加大),全場旗幟方向一致(東南風)——美術即演出。資料來源建議 atmosphere 關鍵幀增加選填 `wind` 欄位(沿用既有插值機制,免新系統)。
  4. `uTime` 同 6-C 用全域 p(確定性);現有 per-unit Mesh 旗幟數量級小,不需 instancing,共用 material 時注意 uniform 共享即可。
- **驗收**:旗面常時微飄;借東風後全場旗幟明顯加劇、方向一致;夜間旗面亮度合理;seek 確定性;FPS 無明顯下降。
